# 龙虾骑士看板模板

## 目标

这套模板用于支撑一个人与多只龙虾 Agent 协作的 SRE 运维服务平台 demo。

适用场景：

- 运维部署与变更任务
- 权限与账号处理任务
- 多 Agent 协同执行的复杂任务
- 需要 Planner 先产出方案、再人工批准、最后人工验收的任务

**任务模板（SPEC）**：对「分析—执行」会多轮来回的复杂任务，在 **人工验收通过** 后，可将可复用的约束与步骤骨架固化为 **SPEC / 提示词规格**，供后续工单在规划中直接引用，减少 Planner 试探成本。定义与示例见 [Planner 任务模板（SPEC）](planner-task-spec-templates.md)。

这套模板刻意保持通用，不绑定 EKS、RDS、GitLab 或其他单一领域。

## 核心原则

1. Planner 负责出方案，不负责直接放行执行。
2. 人工确认通过后，任务才可以进入执行阶段。
3. Agent 执行完成后，必须经过人工验收才能关闭。
4. Agent 不可用、执行失败、验收失败必须分开建模。
5. 流程强约束，字段弱耦合，领域细节放在任务上下文中。

## 标准流程

主流程：

`待办 -> 规划中 -> 待人工确认 -> 待执行 -> Agent执行中 -> 待人工验收 -> 已完成`

异常状态：

- `Agent不可用`
- `执行失败`
- `验收失败`
- `阻塞中`

## 状态说明

| 状态 | 含义 | 负责人 |
| --- | --- | --- |
| 待办 | 任务被提出，尚未开始规划 | 人工 |
| 规划中 | Planner 正在分析任务、拆解步骤、匹配模板与执行龙虾 | Planner |
| 待人工确认 | 方案已产出，等待人工批准是否允许执行 | 人工确认人 |
| 待执行 | 方案已批准，等待调度或目标龙虾领取 | 调度器 / Agent |
| Agent执行中 | 某只龙虾正在执行 | Agent |
| 待人工验收 | Agent 已提交执行结果，等待人工验收 | 验收人 |
| 已完成 | 人工验收通过，任务关闭 | 验收人 |
| Agent不可用 | 目标 Agent 无法执行，如离线、权限不足、环境异常 | 调度器 / 人工 |
| 执行失败 | Agent 已开始执行，但执行过程失败 | Agent / 人工 |
| 验收失败 | Agent 执行完成，但结果不符合预期 | 验收人 |
| 阻塞中 | 外部依赖、审批、资源或条件未满足 | 人工 / Planner |

## 推荐流转规则

### 主路径

- `待办 -> 规划中`
- `规划中 -> 待人工确认`
- `待人工确认 -> 待执行`
- `待执行 -> Agent执行中`
- `Agent执行中 -> 待人工验收`
- `待人工验收 -> 已完成`

### 异常路径

- `待执行 -> Agent不可用`
- `Agent执行中 -> 执行失败`
- `待人工验收 -> 验收失败`
- `任意状态 -> 阻塞中`

### 恢复路径

- `Agent不可用 -> 待执行`
- `执行失败 -> 待人工确认`
- `执行失败 -> 待执行`
- `验收失败 -> 待执行`
- `阻塞中 -> 规划中`
- `阻塞中 -> 待执行`

## 人工关口

### 关口一：待人工确认

确认的是“方案是否允许执行”，而不是“结果是否正确”。

检查重点：

- Planner 输出是否完整
- 执行步骤是否合理
- 风险与回滚方案是否明确
- 目标 Agent 是否匹配
- 前置条件是否满足

### 关口二：待人工验收

验收的是“执行结果是否满足目标”。

检查重点：

- 目标是否达成
- 资源或配置是否符合预期
- 输出物是否完整
- 日志与证据是否可追踪
- 是否需要返工或补充处理

## 任务结构建议

### 父任务

建议用于承载整体目标，例如：

- 创建某套环境
- 开通权限
- 执行某项变更
- 处理某个故障

### 子任务

建议用于承载可执行步骤，例如：

- Planner 子任务
- 单个 Agent 执行子任务
- 验收子任务
- 改进建议子任务

## 最小字段集

这些字段应尽量保持长期稳定，不随具体技术域变化。

| 字段 | 用途 | 备注 |
| --- | --- | --- |
| 任务目标 | 描述这次任务最终要达成什么 | 必填 |
| 任务类型 | 例如部署、变更、权限、巡检、故障处理、优化 | 必填 |
| 服务域 | 例如 AWS、GitLab、Database、Kubernetes、IAM、Network、Other | 必填 |
| 优先级 | 表示处理优先级 | 必填 |
| 风险等级 | Low / Medium / High | 必填 |
| Planner输出 | 记录规划结果、拆解步骤、执行建议 | 必填 |
| 人工确认人 | 谁负责放行执行 | 必填 |
| 人工确认结果 | 待确认、通过、驳回 | 必填 |
| 执行模式 | 人工、单Agent、多Agent | 必填 |
| 执行负责人 | 负责本次执行协调的人或角色 | 建议填写 |
| 执行龙虾 | 主要执行 Agent | 可为空，规划后填写 |
| 候补龙虾 | 主 Agent 不可用时的备选 | 可选 |
| 验收人 | 谁负责验收结果 | 必填 |
| 验收标准 | 如何判断任务完成 | 必填 |
| 当前结果 | 待执行、执行中、成功、失败、待验收等摘要 | 建议填写 |
| 错误摘要 | 失败时记录核心问题 | 可选 |
| 回滚方案 | 失败或验收不通过时的回退策略 | 建议填写 |
| 执行日志链接 | 指向日志、控制台输出或证据 | 可选 |
| 输出物链接 | 指向交付物、结果页、截图或配置证据 | 可选 |
| 任务上下文 | 记录具体领域参数与输入 | 必填 |
| 改进建议 | 是否需要把本次经验回写到模板或流程 | 可选 |

## 任务上下文

为了避免字段过于领域化，具体技术参数不建议做成大量 Jira 自定义字段，而是统一放进 `任务上下文`。

建议用结构化文本记录，例如：

```yaml
system: aws
service: eks
environment: demo
target: lobster-main-cluster
inputs:
  region: ap-northeast-1
  version: "1.31"
  node_count: 3
dependencies:
  - vpc-ready
  - iam-role-ready
notes:
  - planner suggests using reusable provisioning flow
```

如果未来任务切换到 GitLab、RDS、网络、权限或其他平台，只需要改 `任务上下文` 内容，不需要新增一批固定字段。

## 通用模板骨架

建议每个任务都至少包含以下内容：

1. 任务目标
2. 背景
3. 任务上下文
4. Planner 输出
5. 人工确认检查项
6. 执行步骤
7. 失败处理与回滚方案
8. 人工验收标准
9. 执行结果
10. 改进建议

## 状态机摘要

```text
待办
  -> 规划中
  -> 待人工确认
  -> 待执行
  -> Agent执行中
  -> 待人工验收
  -> 已完成

异常分支:
待执行 -> Agent不可用
Agent执行中 -> 执行失败
待人工验收 -> 验收失败
任意状态 -> 阻塞中

恢复分支:
Agent不可用 -> 待执行
执行失败 -> 待人工确认 / 待执行
验收失败 -> 待执行
阻塞中 -> 规划中 / 待执行
```

## 当前结论

这份模板的重点不是把任务领域固定下来，而是把协作规则固定下来：

- 先规划
- 再人工确认
- 再执行
- 再人工验收
- 失败与不可用分开处理

只要遵守这套状态机和最小字段集，这个面板就能承载不确定的未来任务。
